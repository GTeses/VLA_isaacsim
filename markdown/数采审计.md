  ## 2. 采样频率 / 控制频率

  当前 robot_only 环境配置：

  - SIM_DT = 1/120 = 0.008333... s
  - DECIMATION = 2
  - 控制频率 = 120 / 2 = 60 Hz

  因此：

  - HDF5 中每一帧 state/image/action 的记录频率：60 Hz

  也就是说：

  - image/state/action 是一一对应逐 control step 记录
  - 不是低频重采样后的数据

  ## 5. state 语义

  observation/state 是固定 70 维，拼接顺序为：

  state[0:14]   = joint_pos
  state[14:28]  = joint_vel
  state[28:42]  = last_action
  state[42:49]  = left_tcp_pose
  state[49:56]  = right_tcp_pose
  state[56:63]  = object_pose
  state[63:70]  = target_pose

  ### 5.1 joint_pos

  - 维度：14
  - 语义：双臂关节位置
  - 单位：rad
  - 只包含双臂，不包含腰、头、轮子、手指、夹爪

  ### 5.2 joint_vel

  - 维度：14
  - 语义：双臂关节速度
  - 注意：这里不是原始速度，而是环境里已经乘了 JOINT_VELOCITY_SCALE = 0.1
  - 单位可以理解为：rad/s * 0.1

  ### 5.3 last_action

  - 维度：14
  - 语义：上一时刻发送给环境的原始控制 action
  - 单位：无量纲 normalized action
  - 范围：理论上 [-1, 1]

  ### 5.4 pose 段

  - left_tcp_pose / right_tcp_pose / object_pose / target_pose
  - 每段 7 维
  - 格式固定为：

  [x, y, z, qw, qx, qy, qz]

  - 位置单位：m
  - 姿态：四元数

  ———

  ## 6. state 的关节顺序：非常重要

  这里有一个高风险点。

  当前代码历史上已经确认：

  - 环境内部 env._arm_joint_ids 的顺序不是简单的
      - left_joint1 ... left_joint7, right_joint1 ... right_joint7
  - 而是 交错顺序
  - 之前你们已经因此修过一次严重 bug

  当前 state[0:14] 和 actions[0:14] 的顺序，跟随的是：

  - env._arm_joint_ids
  - 也就是环境真实 action/joint 顺序
  - 不是人类直觉上的“左 7 再右 7”

  这意味着：

  - HDF5 里的 14 维 joint/state/action 顺序，大概率是交错的
  - 而不是 left_joint1..7 + right_joint1..7

  非常关键的是：

  - 你们的 robot_only_closing_in_dataset.py 里 ACTION_NAMES 仍然写成了：
      - left_joint1..7
      - right_joint1..7
  - 这和真实 HDF5 顺序可能不一致

  这会直接影响：

  - LeRobot 字段命名
  - OpenPI/pi0.5 action schema
  - 训练时 state/action 对应关系

  ### 结论

  在真正转 LeRobot 前，必须先显式确认 HDF5 14 维顺序，不要只相信 ACTION_NAMES。

  ## 9. action 语义：最关键

  ### 9.1 action 是什么

  当前 actions 不是绝对关节角，也不是下一帧 qpos。

  它是：

  - 14 维 normalized delta-style control action
  - 每维大致在 [-1, 1]
  - 环境收到后会做：

  joint_target = current_joint_pos + action * ARM_ACTION_DELTA_SCALE

  其中：

  - ARM_ACTION_DELTA_SCALE = 0.3

  然后再：

  - clip 到 articulation joint limits

  ### 9.2 也就是说

  action[t] 的真实含义是：

  delta_q_normalized

  而不是：

  - 绝对关节目标 q_target
  - 下一帧 q_{t+1}
  - joint velocity
  - 末端位姿 delta

  ### 9.3 action 是否经过 IK

  是。

  动作链路是：

  structured target
  -> Differential IK
  -> 14D raw action
  -> env 内部转 joint target

  所以数据里的 actions 是：

  - 脚本 IK teacher 输出的 joint-space delta control
  - 不是人工遥操作
  - 不是末端空间 action

  ### 9.4 action 是否 clip / smooth

  有这些处理：

  - 先被 clamp 到 [-1, 1]
  - 再乘 delta_scale=0.3
  - 再 clip 到 joint limits

  没有单独看到额外 low-pass filter。

  ## 13. 图像和动作是否同步

  是同步的，一帧一一对应。

  当前采集逻辑是：

  action = ik.infer(...)
  env.step(action)
  frame = _capture_policy_frame(...)
  episode_actions.append(action)
  episode_states.append(frame["observation/state"])
  episode_images.append(...)

  因此严格来说：

  - action[t] 是这一帧 step 执行时发送的控制
  - state[t] / image[t] 是这个 env.step(action[t]) 之后抓到的 post-step 结果

  所以：

  - image[t] 对应的是 执行了 action[t] 之后的世界状态
  - 不是 pre-action 图像

  这点在转 LeRobot / OpenPI 时要明确。

  ### timestamp

  - 当前没有单独保存 timestamp
  - 但因为控制频率固定 60Hz，可以按固定 dt 推导时间轴

  # 我给分析师的总结判断

  ## 能不能直接转 LeRobot

  技术上可以，但在转之前必须先解决 1 个关键问题：

  - HDF5 里的 14 维顺序，和转换脚本里假定的 joint name 顺序是否一致

  这是当前第一优先级风险。

  ———

  ## action 应该怎么定义

  当前最合理的定义是：

  14D normalized delta joint action

  不是：

  - absolute q_target
  - q_{t+1}
  - velocity
  - end-effector delta pose

  如果要做 LeRobot / OpenPI：

  - action schema 应该明确写成 delta joint command
  - 同时在部署端保持同样的 adapter 语义

  ———

  ## 夹爪维度是否该保留

  当前建议：

  - 第一版不要保留夹爪维度
  - 只做 14DoF 双臂

  因为当前数据里根本没有真实夹爪/灵巧手状态与动作。

  ———

  ## OpenPI / pi0.5 config 怎么写

  第一版建议：

  - 用 pi05_base
  - action_dim = 14
  - observation:
      - external RGB
      - left_wrist RGB
      - right_wrist RGB
      - 70D state
  - task: 单英文 prompt
  - 不要先加夹爪

  但在这之前，要先把：

  - 14 维顺序问题
  - LeRobot 环境安装问题
    解决掉。

  ———

  ## 第一版 LoRA 训练的最大风险

  1. action 14 维顺序可能与命名不一致
  2. action 是 normalized delta，不是绝对角
  3. joint_vel 是缩放过的，不是原始速度
  4. 全部是成功样本，没有失败分布
  5. prompt 只有一个
  6. 视觉域几乎不变
  7. 圆盘不是物理交互体，只是语义 marker pose
